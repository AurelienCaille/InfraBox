Quickstart

Hello World

With InfraBox you can execute a graph of jobs. Each job in the graph can be either a Docker container or multiple container (docker-compose)

In this quickstart quide we will show you some first steps to execute your workflow with InfraBox.

Preparation

  • First you should install infraboxcli:

    pip install infraboxcli

    You can use it to run your jobs on your local machine or push your workflows to InfraBox to offload work from your local machine:

  • Next create a new empty directory and cd into it:

    mkdir tutorial
cd tutorial

  • Let's now create a "hello world" project. We would like to execute a workflow which contains only one job and it should echo "hello world". In InfraBox everything runs in a Docker container. So we first have to create a Dockerfile like this:

    
FROM alpine
CMD echo "hello world"

    The file is quite simple. All the container does is printing "hello world" when it's started.

  • To be able to use InfraBox or the CLI to run your container you additionally have to create a file named infrabox.json in your root directory. It will contain all the information InfraBox requires to execute your workflow. A simple file with one job could look like this: 

    
{
    "version": 1,
    "jobs": [{
        "type": "docker",
        "name": "example1",
        "docker_file": "Dockerfile",
        "build_only": false,
        "resources": { "limits": { "cpu": 1, "memory": 1024 } }
    }]
}

That's all you need. Now simply execute infrabox run and you will see that first the container is build and then executed. To be honest you could have done that as well with docker build and docker run yourself. The next sections will show you some more complicated examples.

Dependencies

InfraBox becomes quite powerful as soon as you need to run multiple jobs which depend on each other. So let's have a quick look at a producer/consumer example in which one job produces a file and another job reads the file.

Example

Now we need two Dockerfiles, one for the producer (Dockerfile_producer):


FROM alpine
CMD echo "hello world" > /infrabox/output/data.txt

And one for the consumer (Dockerfile_consumer):


FROM alpine
CMD cat /infrabox/inputs/producer/data.txt

As you can see the producer writes the string "hello world" into a file /infrabox/output/data.txt. The directory /infrabox/output is special. Every file a job writes into this directory can be accessed by dependent jobs. The data will be accessible by the dependend jobs at /infrabox/inputs/<job_name>/. For more details how dependencies work see here.

Last thing to do is to tell InfraBox the dependency between the two jobs. For this we create the infrabox.json like this:


{
    "version": 1,
    "jobs": [{
        "type": "docker",
        "name": "producer",
        "docker_file": "Dockerfile_producer",
        "build_only": false,
        "resources": { "limits": { "cpu": 1, "memory": 1024 } }
    }, {
        "type": "docker",
        "name": "consumer",
        "docker_file": "Dockerfile_consumer",
        "build_only": false,
        "resources": { "limits": { "cpu": 1, "memory": 1024 } },
        "depends_on": ["producer"]
    }]
}

As you can see we now have two jobs defined and the consumer depends on the producer. With this you can already model quite complex workflows in which every job can be its own container.

If this worked without any problems, you're good to go to create your first InfraBox project.

A simple Node.js application

Now that we have a working version of infraboxcli installed we can continue and create a simple Node.js app. We'll use this only for demo purposes, so it will not have much functionality. To run the code locally make sure you have Node.js installed.

In an empty directory create the file package.json. It contains the relevant information about the Node.js project as well as the dependencies for running. Save the following to package.json.


{
  "name": "infrabox-demo",
  "version": "1.0.0",
  "description": "The project will set up a very simple web server and we will execute some very basic tests.",
  "main": "index.js",
  "scripts": {
    "test": "mocha spec.js"
  },
  "author": "",
  "license": "MIT",
  "dependencies": {
    "express": "^4.15.2",
    "supertest": "^3.0.0"
  },
  "devDependencies": {
    "mocha": "^3.3.0"
  }
}

Now run npm install. This will download all the required dependencies and save them to the node_modules directory.

Next step is to write the actual application. Create a file called index.js and copy & paste the below code:


var express = require('express');
var app = express();

app.get('/', function (req, res) {
    res.send('Hello World!');
});

var server = app.listen(8080, function () {
    console.log('Example app listening on port 8080!');
});

module.exports = server;

The example contains a very simple web server which just returns "Hello World!". If you now run node index.js it should output: 


Example app listening on port 8080!

You may now open your browser and go to http://localhost:8080. You should now see the text "Hello World!". Congratulations, you have set up a simple Node.js application. Next we will add some tests to check if our web server really works the way we expect it. Create the file spec.js in the root folder of your project and paste this code:


var request = require('supertest');

describe('loading express', function () {
  var server;
  beforeEach(function () {
    server = require('./index');
  });

  afterEach(function () {
    server.close();
  });

  it('responds to /', function testSlash(done) {
      request(server)
        .get('/')
        .expect(200, done);
  });

  it('404 everything else', function testPath(done) {
    request(server)
      .get('/foo/bar')
      .expect(404, done);
  });
});

The tests simply start the server and check whether the server returns the status 200 for "/" and 404 for "/foo/bar". You can now run the tests with npm test. Both tests should succeed.

Create a Dockerfile

Next we’re going to create a Dockerfile. InfraBox uses Docker for all jobs. So for everything you want to run on InfraBox you need at least a very simple Dockerfile. If you're already using Docker for your projects you probably need no or only minor adjustments to run them on InfraBox.

If you haven't used Docker containers yet you should have a look at the Docker documentation. For your first steps especially the Docker Getting Started Guide will be very helpful to understand how to set up your project using containers.

Now create an directory infrabox and in this directory create the file Dockerfile and paste this code into it:


# Use nodejs 6
FROM node:6-alpine

# Copies our code into the container
COPY . /project
WORKDIR /project

# Entrypoint of the container
ENTRYPOINT /project/infrabox/entrypoint.sh

As you see in the above Dockerfile we set the entrypoint of the container to a entrypoint.sh. This script will be executed when the container is started by InfraBox. So everything you want to run should be placed in this file. For your example application create the file entrypoint.sh in the infrabox directory next to the Dockerfile like this:


#!/bin/sh
set -e

echo "## Install dependencies"
export NPM_CONFIG_LOGLEVEL=warn # to reduce log output
npm install

echo "## Run tests"
npm test

The script must be executable, so make sure it has the propper permissions and run chmod +x infrabox/entrypoint.sh.


docker build  -f infrabox/Dockerfile -t test-app .

And then running it:


docker run test-app

The output should be similar to:


## Install dependencies
npm WARN infrabox-demo@1.0.0 No repository field.
## Run tests

> infrabox-demo@1.0.0 test /project
> mocha spec.js

  loading express
Example app listening on port 8080!
    ✓ responds to /
    ✓ 404 everything else

  2 passing (120ms)

As you can see the script installs our dependencies and starts the tests. You can try this yourself locally by first building the container.

Now we have a simple Node.js application which we validate with our simple Mocha tests in a Docker container. Next is to set up the project for using infraboxcli.

Run it with infraboxcli

Now as we can run our project within a Docker container we can use InfraBox to run it for you. As InfraBox needs to know which Docker container to run (you may have multiple in the same project), you have to create a file called infrabox.json in the root directory of your project. It contains all the information about which Docker containers should be run and what their dependencies are. We will talk about job dependencies in a later part. For now paste this code into the file:


{
    "version": 1,
    "jobs": [{
        "name": "tests",
        "type": "docker",
        "resources": { "limits": { "cpu": 1, "memory": 1024 } },
        "docker_file": "infrabox/Dockerfile",
        "build_only": false
    }]
}

For some more details about the structure of the infrabox.json and all the possible options see here.

That's all you need. You may now run infrabox list to list all the jobs in your project. As we only have one job set up it should only show you "tests".

Note: You can simply use infrabox run to run your jobs. It will build the container we created above and run it. See here for more information about how to run jobs locally.

Push to InfraBox

In the previous step we have set up a local project and used infraboxcli to run it. Now it's time to make use of the InfraBox infrastructure and run the project there. In general there are two possibilities to do this. You can either create a GitHub repository and connect it to InfraBox. Every time you push to the repository InfraBox will get notified about new changes, checkout your code and run all your jobs. See here on how to set up your project with Github. In this tutorial we use the second option without a GitHub Repository. You can simply use infraboxcli to push your jobs to InfraBox. The CLI will zip your local directory, upload it to InfraBox and execute all your jobs defined in infrabox.json.

To be able to push your jobs you first have to create a project in the InfraBox Dashboard. See here on how to do it. Additionally you have to create an Authentication Token which the CLI will use to authenticate your requests. This prevents other people from pushing to your project. You can revoke the tokens at any time. See here on how to create them.

So if you have your project and Authentication Token created you only need to run:

export INFRABOX_API_URL=<API_URL>
export INFRABOX_CLI_TOKEN=<YOUR_AUTH_TOKEN>
infrabox push

The API_URL of infrabox.ninja playground is https://infrabox.ninja/api/cli.

That's it, you can now offload your jobs to the InfraBox infrastructure. Go to the Dashboard and see your job running. You may also use --show-console to print the console output of your jobs. See here for more details.

Use the cache

InfraBox offers you the possibility to cache some files between runs. This can be very useful for data which changes rarely and is time consuming to install. In our example our node_modules directory is a perfect fit. So we don't have to wait every run to download and install all dependencies.

InfraBox mounts a cache directory /infrabox/cache into every container. Every data which is copied into this directory will be available in the next run of the job. See here for some more details.

Make sure you put your node_modules directory into the .dockerignore and .gitignore files. Or else you may overwrite the cache with your local files.

We modify our infrabox/entrypoint.sh file to make use of the cache: 

#!/bin/sh
set -e

cd project

# Here we load the cached node_modules directory
echo "## Prepare cache"
mkdir -p /infrabox/cache/node_modules
cp -r /infrabox/cache/node_modules /project

echo "## Install dependencies"
export NPM_CONFIG_LOGLEVEL=warn # to reduce log output
npm install

# Copy updated node_modules back to cache
echo "## Copy to cache"
rm -rf /infrabox/cache/node_modules
cp -r /project/node_modules /infrabox/cache/node_modules

echo "## Run tests"
npm test

As you see we simply copy our node_moules directory to /infrabox/cache after we installed it. So we don't have to download all our npm dependencies again in the next run. See here for more details on caching.

Multi Container Tests

So far we started our web server within our tests. So we had one docker container which contains the tests and the server. This image we would usually not deploy to our production infrastructure, because there's not good reason to put test code and all the test dependencies into the container.

To solve the problem we can split our container into two different containers. One which contains only our server and the second one the tests. The first container can now be set up in a way it would also run on a production infrastructure. This way we really test what we run later on.

But for testing it we now have to start two containers. For this InfraBox offers the possibility to run docker-compose jobs. With this you can create a docker-compose file and run it like one job. See TODO for more details.

First we create a second Dockerfile in the root directory of our project. With this we build our image for the server without any tests. This image could later also be used in a production environment and should therefore contain only minimal dependencies to run the server.


# Use nodejs 6
FROM node:6-alpine

# Copy only necessary files
COPY package.json /project/package.json
COPY entrypoint.sh /project/entrypoint.sh
COPY index.js /project/index.js
WORKDIR /project

# Entrypoint of the container
ENTRYPOINT /project/entrypoint.sh

For running the web server we only need a few files. Be careful, we also use a second entrypoint.sh for starting a server. This file should be located in the root directory of the project and look like this (we did not add caching to keep the example simpler):

#!/bin/sh
set -e

echo "## Install dependencies"
export NPM_CONFIG_LOGLEVEL=warn # to reduce log output
npm install

echo "## Start web server"
node index.js

Now we have got a container which starts our web server, but we also need a second container which contains the tests. We use infrabox/entrypoint.sh as the entrypoint for our tests container. For this we'll modify it slightly (we remove the caching to simplify the example):

#!/bin/sh
set -e

echo "## Install dependencies"
export NPM_CONFIG_LOGLEVEL=warn # to reduce log output
npm install

# Wait until the web server is up and running before we start our tests
until $(curl --output /dev/null --silent --head --fail http://server:8080/); do
  >&2 echo "Web server host is unavailable - sleeping"
  sleep 1
done

echo "## Run tests"
npm test

As we now run the web server in a separate container we have to wait until the container has started up and the server is able to serve requests. After this is the case we can start our tests.

Obviously we also have to modify our tests to connect to the web server instead of starting the server in the tests. The modified tests in spec.js now look like this:

var request = require('supertest');
var server = request.agent('http://server:8080')

describe('loading express', function () {
  it('responds to /', function testSlash(done) {
    server
	  .get('/')
      .expect(200, done);
  });

  it('404 everything else', function testPath(done) {
    server
      .get('/foo/bar')
      .expect(404, done);
  });
});

With the previous changes we now have two containers, one for the web server and one for the tests. Last step is to bring them together with the help of docker-compose. For more details about docker-compose please look here. Create a file infrabox/docker-compose.yml like this:

version: "3.2"

services:
    server:
        build:
            context: ..
            dockerfile: Dockerfile
    test:
        build:
            context: ..
            dockerfile: infrabox/Dockerfile
        links:
            - server

As you can see we use our two Dockerfiles for the server and the tests. The test service also needs a link to the server so the tests are able to access the server at http://server:8080.

We are almost done. Last step is to modify infrabox.json in a way that it does not start our old docker container anymore, but now uses docker-compose to run both our containers.

{
    "version": 1,
    "jobs": [{
        "name": "tests",
        "type": "docker-compose",
        "resources": { "limits": { "cpu": 1, "memory": 1024 } },
        "docker_compose_file": "infrabox/docker-compose.yml"
    }]
}

We changed the type of the job to docker-compose and instead of docker_file we use docker_compose_file.

This is it. We can now run our tests again with either infrabox run on our local machine or with infrabox push on the InfraBox infrastructure.

Workflow Configuration

If you want to run jobs on InfraBox you have to create a file called infrabox.json in the root directory of your project. It tells InfraBox what jobs are part of the project and which dependencies between the jobs exist. The basic structure of the file is as follows:

{
    "version": 1,
    "jobs": [
        ...
    ]
}
Name Required Type Description
version true number Currently only valid value is 1.
jobs true array of job types A list of the actual jobs to run. It can be either
  • docker
  • docker-compose
  • git
  • workflow

docker

The job type docker is one of the most important jobs. You can use it to run any Docker container you like.

{
    "version": 1,
    "jobs": [{
        "type": "docker",
        "name": "build",
        "docker_file": "infrabox/build-and-test/Dockerfile",
        "resources": { "limits": { "cpu": 1, "memory": 1024 } },
        "build_only": false,
        "depends_on": ["other_job_name"],
        "environment": { ... },
        "build_arguments": { ... }
    }]
}
Name Required Type Description
type true string Has to be "docker" to run a single Docker container
name true string Name of the job
docker_file true string Path to the Dockerfile
resources.limits.cpu true integer Specify the number of CPUs to be used for the job. Value must be an integer.
resources.limits.memory true integer Specify the number of MiB of RAM to be used for the job. Value must be an integer and at least 128.
timeout false integer Timeout in seconds after which the job should be killed. Timeout starts when the job is set to running. Default is 3600 (1 hour).
build_only true boolean If set to true the container will only be build but not run. Use it if you only want to build a container and push it to a registry. See here for how to push to a docker registry.
depends_on false array of job names Jobs may have dependencies. You can list all jobs which should finish before the current job may be executed.
environment false object Can be used to set environment variables for the job. See Environment Variables for more details.
build_arguments false object Can be used to set docker build arguments. See Build Arguments for more details.

When InfraBox builds a docker container the docker context is always the directory in which the infrabox.json is stored. So when you copy files in your Dockerfile the path must be relative to the infrabox.json file.

Suppose we have the following files in our project:

/project/
  • src/
    • Dockerfile
    • test.py
  • infrabox.json

And our Dockerfile looks like this:

FROM alpine:3.4
COPY src/test.py /usr/bin/test.py
CMD /usr/bin/test.py

As you can see the COPY uses src/test.py and is therefore relative to the root directory of the project and not the Dockerfile.

docker-compose

Sometimes it's useful to run multiple Docker containers, i.e. if you want to test your web application you may want to start the database, web server and the actual tests in separate Docker containers. For this docker-compose jobs can be used. Instead of specifying only one Docker container you can give an entire docker-compose file.

The job ends as soon as one of the container stops. If it stops with an error code other than 0 the job is considered to be failed, else it finished succesfully.

An example docker-compose job definition:

{
    "version": 1,
    "jobs": [{
        "type": "docker-compose",
        "name": "test",
        "docker_compose_file": "infrabox/test/docker-compose.yml",
        "resources": { "limits": { "cpu": 1, "memory": 1024 } },
        "depends_on": ["other_job_name"],
        "environment": { ... }
    }]
}
Name Required Type Description
type true string Has to be "docker-compose" to run multiple containers
name true string Name of the job
docker_compose_file true string Path to the docker-compose.yaml
resources.limits.cpu true integer Specify the number of CPUs to be used for the job. Value must be an integer.
resources.limits.memory true integer Specify the number of MiB of RAM to be used for the job. Value must be an integer and at least 128
timeout false integer Timeout in seconds after which the job should be killed. Timeout starts when the job is set to running. Default is 3600 (1 hour)
depends_on false array of job names Jobs may have dependencies. You can list all jobs which should finish before the current job may be executed. See here for more details about using dependencies
environment false object Can be used to set environment variables for the job. See Environment Variables for more details.

git

It's not necessary to have all your job definitions in your own project. With the git job type you can include job definitions for any accessible git repository.

An example git job definition:

{
    "version": 1,
    "jobs": [{
        "type": "git",
        "name": "some-external-jobs",
        "git_url": "https://github.com/InfraBox/examples.git",
        "commit": "master",
        "depends_on": ["other_job_name"]
    }]
}
Name Required Type Description
type true string Has to be "git" to include a workflow from a repository
name true string Name of the job
clone_url true string The url used to clone the repository.
commit true string Can be any identifier which git can checkout. So a branch name, commit sha1 or tag.
depends_on false array of job names Jobs may have dependencies. You can list all jobs which should finish before the current job may be executed. See here for more details about using dependencies
environment false object Can be used to set environment variables for the job. See Environment Variables for more details.

workflow

If you have many jobs you might want to structure them in different files.

An example job definition:

{
    "version": 1,
    "jobs": [{
        "type": "workflow",
        "name": "deploy",
        "infrabox_file": "deployments.json",
        "depends_on": ["test"]
    }]
}
Name Required Type Description
type true string Has to be "workflow" to include a workflow from a repository
name true string Name of the job
infrabox_file true string Path to another infrabox.json file in the local project
depends_on false array of job names Jobs may have dependencies. You can list all jobs which should finish before the current job may be executed. See here for more details about using dependencies
environment false object Can be used to set environment variables for the job. See Environment Variables for more details.

Job Dependencies

To model dependencies between jobs you simply use depends_on in the job description. As an example we create tree jobs A, B and C. C depends on A and B.


{
    "version": 1,
    "jobs": [{
        "name": "A",
        ...
    }, {
        "name": "B",
        ...
    }, {
        "name": "C",
        ...
        "depends_on": ["A", "B"]
    }]
}

Jobs A and B would start in parallel and C would only start if A and B succeeded successfully. If either of the parent jobs failed C would not be started at all. You may use dependencies with all available job types.

Transfer data between jobs

If you have defined dependencies between your jobs you can also transfer data between them. The parent job may produce some output which the child job may access.

Every InfraBox job has automatically an output folder mounted under /infrabox/output. All data which is written into this directory will be accessible by the direct child jobs. The data is made accessible in the child in the /infrabox/inputs/<parent_job_name> directory.

Suppose we have the following job definition


{
    "version": 1,
    "jobs": [{
        "name": "parent",
        ...
    }, {
        "name": "child",
        ...
        "depends_on": ["parent"]
    }]
}

If the the parent job would create the file /infrabox/output/hello_world.txt then the child job would be able to access it under /infrabox/inputs/parent.

Environment Variables

With InfraBox you have two ways of setting environment variables for your jobs. The easiest way is to add it to your job definition like this:

{
    "type": "docker",
    ...
    "environment": {
        "ENV_VAR_1": "some value",
        "ENV_VAR_2": "some other value"
    }
}

You may use this approach to store values which don't contain and secret information like passwords, because everybody could read it in you repository.

If you want to store secret values you should add them first in the InfraBox Dashboard:

  • Login to the InfraBox Dashboard
  • Select the project for which you wan to set a secrets
  • Navigate to the "Settings" tab
  • In the "Secrets" section insert a name and value
  • Click on the "+" sign

With this you now have created a secret value which you can use in your jobs without writing the value into a text file.

To use it in your job reference it in your job definition like this:

{
    "type": "docker",
    ...
    "environment": {
        "ENV_VAR_1": { "$secret": "NAME_OF_SECRET" }
    }
}

In your job the secret value will then be available as the environment variable ENV_VAR_1.

Build Arguments

Sometimes you have to set variable for the docker build process. You can set docker build arguments like this: 

{
    "type": "docker",
    ...
    "build_arguments": {
        "ARG1": "some value",
        "SOME_OTHER_ATH": "some other value"
    }
}

Using Image Registries

The docker images which have been build can also be pushed to a docker registry by InfraBox. You would add a deployment to your job: 

{
    "type": "docker",
    ...
    "deployments": [{
            "type": "docker-registry",
            "host": "<my_registry>:5000",
            "repository": "repo-name",
            "username": "your_username",
            "password": { "$secret": "SECRET_NAME_FOR_YOUR_PASSWORD" }
    }]
}

Deployments are currently only supported for docker job types. username and password are optional if you don't have any authentication setup for your registry.

Caching on InfraBox

Sometimes it's useful to keep some data from one run of a container to the next one. Maybe you have a nodejs project and don't want to install your dependencies every time. For such uses cases InfraBox mounts the directory /infrabox/cache into every container. Everything which you store in this directory will be available at the same place in the next run. So for your nodejs project you could simply copy your node_modules directory in there.

Create Badges

With InfraBox you can easily create one or mutiple badges like or with your custom information. To create a badge simply create a json file like this: 

{
    "version": 1,
    "subject": "build",
    "status": "failed",
    "color": "red"
}
and save it to /infrabox/upload/badge/badge.json. You can also save multiple files with different names in this directory. For every file a separate badge will be created.

Upload Test Result

InfraBox can show your testresult on a tab at the job detail view. This gives you the possibility to quickly check the results and if one of your tests has failed you can even see the error message and stacktrace if available. To use this feature you only have to copy your testresult file to /infrabox/upload/testresult. We will try to auto detect the format of your file. Currently supported file formats are:

  • xunit (xml)

Upload Coverage Result

InfraBox can show your coverage results on a tab at the job detail view. To use this feature you only have to copy your coverage file to /infrabox/upload/coverage. We will try to auto detect the format of your file. Currently supported file formats are:

  • cobertura (xml)
  • clover (xml)

Dynamic Workflows

Often a pre-defined static workflow is not sufficient to handle all use cases. For this InfraBox supports dynamic workflows. Every job in your workflow may produce an infrabox.json in its output directory /infrabox/output/infrabox.json. All jobs defined in there will be automatically added as predecessor jobs. This give you the options to extend your workflow at runtime.

This feature is extremely useful for bigger projects. Maybe you don't want to run certain tests on a branch or you want to additionaly deploy your containers to an registry if you create a tag.

Create an Upload Project

Upload project can be used for executing workflows without storing them in a git repository.

  • Login to the InfraBox Dashboard
  • On the "Overview" page click on "+" on the bottom right corner
  • Select upload and click continue
  • Enter a name and click continue
  • Select if your project should be visible by everybody (public) or only collaborators (private). Click on continue.

Connect a Github Repository

Connecting a Github Repository is very simple.

This is only possible if your InfraBox installation has been setup to work either with github.com or a GitHub Enterprise installation.
  • Login to the InfraBox Dashboard
  • On the "Overview" page click on "+" on the bottom right corner
  • Select GitHub and click continue
  • Enter a name for your project
  • Click on the repository you want to connect and click continue
  • Select if your project should be visible by everybody (public) or only collaborators (private). Click on continue.

Now every push to your github repository will automatically trigger a build.

Create Auth Token

Auth tokens are used by infraboxcli and are like passwords. So make sure you store them at a secure place.

  • Login to the InfraBox Dashboard
  • Select the project for which you want to create a token
  • Go to the settings tab
  • In the Tokens section enter a description and click "+"
  • You will see the token in a popup. You will not be able to retrieve the token again, so make sure you copy it to a secure place

You may use the token to push workflows or pull jobs
